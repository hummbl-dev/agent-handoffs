# v0.1 Scope and Boundary

This document defines the scope, non-goals, adoption boundary, review expectations,
and non-canon posture for v0.1 of the agent-handoffs candidate namespace. It is the
authoritative reference for what v0.1 is, what it is not, and how it may be used.

## Status

Candidate. Not a standard. Not adopted guidance. v0.1 is an exploratory baseline
intended to make agent handoffs version-controlled, reviewable, testable, auditable,
and agent-operable. Nothing in v0.1 is canon unless a later reviewed governance path
adopts it.

## Scope

v0.1 covers the minimal, executable artifacts needed to describe a single agent
handoff as a reviewable packet:

- A **handoff packet** schema (`schemas/agent-handoffs-v0.1.json`) describing one
  handoff from a source agent (or human) to a target agent (or human).
- A **context envelope** carrying the minimal state required for the receiving agent
  to continue work: objective, in-scope items, out-of-scope items, constraints, and
  pointers to prior context.
- A **delegation scope** stating what authority is being transferred, for how long,
  and under what limits.
- A **receipt chain** recording provenance for the packet so handoffs remain
  auditable across hops.
- **Examples and fixtures** that demonstrate valid and invalid packets so the schema
  is testable, not just descriptive.
- **Prior-art references** (`docs/prior-art.md`) placing v0.1 inside the existing
  ecosystem rather than claiming novelty over it.

v0.1 is intentionally single-packet and single-hop oriented. Multi-hop orchestration,
long-running sessions, and runtime negotiation are acknowledged as future work but
are not modeled by the v0.1 schema.

## Non-goals

v0.1 does **not** attempt to:

- Define a universal standard for agent handoffs.
- Claim ownership of the broader field, terminology, or any adjacent protocol.
- Canonize HUMMBL, BaseN, Ownward, or any private/internal concepts unless
  explicitly marked and audited through a reviewed governance path.
- Model full multi-agent orchestration, scheduling, or runtime task allocation.
- Specify a transport protocol, wire format, or message bus.
- Mandate a specific identity, authentication, or key-management system.
- Include private, internal, or secret operational content.

## Adoption boundary

v0.1 artifacts are **candidate** material. They may be referenced, copied, forked,
and experimented with, but they must not be presented as adopted guidance. Any
downstream use must:

1. Mark the material as candidate status.
2. Link back to this repository and to this boundary document.
3. Avoid implying endorsement by this repository or its contributors.
4. Keep public/private boundaries intact; do not import private operational content
   into derivative work based on this repo.

Adoption of v0.1 patterns into any production or governance path requires a separate
review process that is out of scope for this repository at v0.1.

## Review expectations

All v0.1 artifacts are expected to be small, reviewable, and evidence-backed. Reviewers
should expect:

- Schema changes to ship with valid and invalid fixtures that demonstrate the change.
- Examples to avoid private operational content and to mark candidate status clearly.
- Prior-art references to link to public documentation, not to private material.
- Provenance to be preserved: every packet carries a receipt chain; every artifact
  in this repo carries a receipt in `receipts/`.

Reviewers should reject contributions that canonize private concepts, claim ownership
over terminology, or import secret operational content.

## Non-canon posture

This repository is explicitly **non-canon**. v0.1 is a candidate baseline, not a
governing reference. The vocabulary used here (handoff, delegation, context transfer,
agent-to-agent, session continuity, state handoff, context envelope, delegation
token, state transfer, receipt chain, authority propagation) is borrowed from and
shared with the broader ecosystem documented in `docs/prior-art.md`. No term is
claimed as proprietary, and no definition here is asserted as authoritative for the
field.

## Required packet pieces

A v0.1 handoff packet must contain, at minimum:

1. `schemaVersion` — the schema version this packet conforms to.
2. `packetStatus` — lifecycle status of the packet (e.g. `draft`, `issued`, `accepted`).
3. `handoffManifest` — identifying metadata for the packet itself: `id`, `title`,
   `version`, and `scope`.
4. `authority` — who or what issued the packet and under what authority.
5. `handoffContract` — the core handoff agreement:
   - `fromAgent` — the source of the handoff.
   - `toAgent` — the intended recipient.
   - `contextEnvelope` — the minimal state being transferred.
   - `delegationScope` — what authority is delegated and its limits.
   - `expiry` — when the delegated authority and packet lapse.
   - `receiptChain` — provenance entries for this packet and prior hops.
6. `receiptRequirements` — what the receiving agent must produce to acknowledge and
   close the handoff.

Packets missing any of these pieces are invalid against the v0.1 schema. See
`fixtures/valid/` and `fixtures/invalid/` for executable examples.
